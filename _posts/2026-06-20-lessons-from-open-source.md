Contributing to open source changed how I write software. Here are five lessons from the trenches.

## 1. Read the Code, Not Just the Docs

Documentation is often incomplete or outdated. The real truth is in the source code. Learning to read unfamiliar codebases quickly is a superpower that open source forces you to develop.

## 2. Good Issues Make Good Contributions

A well-written issue is worth its weight in gold. Include minimal reproduction steps, expected vs. actual behavior, and the versions of everything involved. The more work you do upfront, the faster maintainers can help.

## 3. Small PRs Win

Nobody wants to review a 2,000-line pull request. Break your changes into the smallest logical units. Each PR should do exactly one thing. Your code will be reviewed faster and with better attention to detail.

## 4. You Are Not Your Code

Code reviews are critiques of the work, not the person. Learn to separate your ego from your pull requests. When a maintainer asks for changes, they're helping you, not attacking you.

## 5. It's About People, Not Code

The best open source projects are communities first, codebases second. Take time to be helpful in issues, write kind review comments, and thank contributors. Building software is ultimately a social activity.

Open source taught me more about software engineering than any course or book ever did. Start small, be patient, and keep showing up.
